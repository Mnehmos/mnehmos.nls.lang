    Quicksort — An Honest Review¶

    What happens when you implement a real algorithm in a specification language?

    The Challenge¶

    A Reddit user in r/vibecoding asked the right question:

    "Could you give me an example of implementing quicksort in NLS? I'd love to see how English would describe that without resorting to just writing Python or without being insanely verbose."

    Fair challenge. Quicksort has recursion, partitioning, list comprehensions — it exercises the parts of NLS that matter. We ran the experiment honestly. Here's everything: what works, what breaks, what the compiler actually produces.

    The Spec¶

    @module sorting
@target python

[quick-sort]
PURPOSE: Sort a list of numbers using the quicksort algorithm
INPUTS:
  - items: list of number
EDGE CASES:
  - len(items) < 2 -> return items
LOGIC:
  1. items[0] -> pivot
  2. [x for x in items if x < pivot] -> lesser
  3. [x for x in items if x == pivot] -> equal
  4. [x for x in items if x > pivot] -> greater
  5. [quick-sort](lesser) -> sorted_lesser
  6. [quick-sort](greater) -> sorted_greater
RETURNS: sorted_lesser + equal + sorted_greater

@test [quick-sort] {
  quick_sort([3, 1, 4, 1, 5]) == [1, 1, 3, 4, 5]
  quick_sort([]) == []
  quick_sort([1]) == [1]
  quick_sort([5, 4, 3, 2, 1]) == [1, 2, 3, 4, 5]
}

    24 lines. Each LOGIC step is precise enough for the compiler to emit real code. The [quick-sort](lesser) syntax is an ANLU reference: it tells the compiler "call this function recursively."

    The Same Structure in Japanese¶

    The point of NLS is not "English, but with nicer comments." The point is that the structure is universal while the surface language is yours.

    In this version, the NLS source removes the English control words from the surface syntax too: no for, no if, no return, and no len.

    @モジュール sorting_ja
@ターゲット パイソン

[クイックソート]
目的: 数値のリストをクイックソートで並べ替える
入力:
  - 項目: 数値のリスト
エッジケース:
  - 長さ(項目) < 2 -> 返す 項目
ロジック:
  1. 項目[0] -> ピボット
  2. [要素 を 項目 から もし 要素 < ピボット] -> 小さい項目
  3. [要素 を 項目 から もし 要素 == ピボット] -> 等しい項目
  4. [要素 を 項目 から もし 要素 > ピボット] -> 大きい項目
  5. [クイックソート](小さい項目) -> 整列済みの小さい項目
  6. [クイックソート](大きい項目) -> 整列済みの大きい項目
返り値: 整列済みの小さい項目 + 等しい項目 + 整列済みの大きい項目

@テスト [クイックソート] {
  クイックソート([3, 1, 4, 1, 5]) == [1, 1, 3, 4, 5]
  クイックソート([]) == []
  クイックソート([1]) == [1]
  クイックソート([5, 4, 3, 2, 1]) == [1, 2, 3, 4, 5]
}

    Same algorithm. Same dependency structure. Same target language. Only the surface syntax changes.

    What the Compiler Produces¶

    $ nlsc compile examples/sorting.nl

  [OK] Parsed 1 ANLUs
  [OK] Resolved dependencies
  [OK] Generated sorting.py (28 lines)
  [OK] Generated test_sorting.py
  [OK] Updated sorting.nl.lock

    Three outputs from one spec. Let's look at each.

    sorting.py¶

    """
Generated by nlsc from examples/sorting.nl
Module: sorting
"""

from __future__ import annotations


def quick_sort(items: list[float]) -> list[float]:
    """
    Sort a list of numbers using the quicksort algorithm

    Args:
        items: list of number

    Returns:
        sorted_lesser + equal + sorted_greater
    """
    if len(items) < 2:
        return items
    pivot = items[0]
    lesser = [x for x in items if x < pivot]
    equal = [x for x in items if x == pivot]
    greater = [x for x in items if x > pivot]
    sorted_lesser = quick_sort(lesser)
    sorted_greater = quick_sort(greater)
    return sorted_lesser + equal + sorted_greater

    This is working Python. Not a scaffold. Not TODO stubs. Run it:

    $ python -c "from sorting import quick_sort; print(quick_sort([3, 1, 4, 1, 5]))"
[1, 1, 3, 4, 5]

    What the compiler produced from the spec:

    • Edge case compiled to an early return guard
    • Each LOGIC step compiled to a Python assignment — list comprehensions, indexing, and recursive ANLU calls all emitted directly
    • RETURNS expression compiled to the final return statement
    • Typed signature, docstring, and module header derived from the spec metadata

    The important point is not that Python can contain Japanese identifiers. The important point is that the source spec no longer forces English control words at all.

    How LOGIC Steps Compile¶

    NLS LOGIC steps use the -> binding syntax: the description before the arrow is what to compute, the name after is what to call the result. When the description is a valid Python expression, the compiler emits it directly:

    NLS LOGIC step Generated Python
    items[0] -> pivot pivot = items[0]
    [x for x in items if x < pivot] -> lesser lesser = [x for x in items if x < pivot]
    [quick-sort](lesser) -> sorted_lesser sorted_lesser = quick_sort(lesser)

    The [quick-sort](args) syntax is an ANLU reference — it calls another function defined in the same module, converting kebab-case to snake_case automatically.

    When a LOGIC description is pure English (like "Calculate the weighted average"), the compiler emits a named variable with a TODO comment. The spec is still useful as documentation; the implementation gets filled in separately.

    Compare to raw Python

    How would you get the same audit surface from writing Python directly? You'd need to write the docstring, the type hints, the tests, and the implementation all separately — and hope they stay in sync. NLS derives all of them from a single source of truth.

    Known Limitations¶

    We ran into real issues during this experiment. Here they are.

    Guards require Python expressions

    Natural language guards don't compile:

    # FAILS:
GUARDS:
  - items must not be empty -> ValueError("Cannot search empty list")

    Guards are verification gates — they need to be machine-evaluable. Use explicit Python expressions:

    # WORKS:
GUARDS:
  - len(items) > 0 -> ValueError("Cannot search empty list")

    EDGE CASES require Python conditions

    Same rule as guards. len(items) < 2 compiles; items has fewer than 2 elements becomes a comment. The condition needs to be something the Python runtime can evaluate.

    LOGIC conditionals

    Natural language conditions in LOGIC steps don't compile to executable code:

    # Compiles but "equals" is not valid Python:
LOGIC:
  5. IF items[mid] equals target THEN return mid

    Use Python operators: IF items[mid] == target THEN return mid.

    Tree-sitter grammar

    The tree-sitter parser requires a compiled grammar .so that isn't built during pip install nlsc[treesitter]. The grammar source exists in tree-sitter-nl/ but needs manual compilation. is_available() returns False even with tree-sitter installed.

    The Trade¶

    Here's the honest comparison.

    24 lines of NLS spec that anyone on the team can read — product managers, QA, new hires — and now read in their own language — versus 12 lines of Python that still inherits English-only syntax.

    From those 24 lines, the compiler gives you:

    • Working Python implementation
    • Typed function signature with docstring
    • Edge case handling as early returns
    • 4 executable pytest tests
    • A lockfile with cryptographic hashes for provenance

    From the 12 lines of Python, you get: a working sort function. Tests, docs, type annotations, and localization are your problem.

    The spec is the contract. The generated code is the artifact. Change the spec, recompile, and everything stays in sync.

    @literal escape hatch

    If you need exact implementation control, NLS provides @literal blocks that bypass the compiler and emit code exactly as written. This is a last resort for performance-critical code, complex regex, or cases where the spec-to-scaffold workflow doesn't apply. See the Language Specification for syntax details.
